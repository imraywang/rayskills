# ray-thread：实战经历 → Thread 装配骨架

一条好的 build-in-public thread 不是写出来的,是**从真实经历里挖出来的**。这个 skill 干挖掘和装配的粗活;定调的话、观点的锋芒,作者自己上。

## 铁律:不代笔

输出永远是**骨架 + 素材**,不是成品:

- 事实、数字、时间线、代码/命令、截图位——填实
- 观点句、判断句、情绪句——留 `【你的话:____】` 空位,并附一行"这里读者在等什么"的提示
- 禁止生成任何以作者第一人称口吻的观点陈述

违反这条,整个输出作废。

## 输入

三种都接:

```
/ray-thread 上周排查并清掉了几台服务器上的挖矿木马,想发条 thread
/ray-thread ~/projects/xxx/RETRO.md          # 复盘文档
/ray-thread (无参数)                          # 追问最近有什么值得讲的实战
```

信息不足时追问,一次问齐:这事的起点是什么?最坏的时刻?最后的数字结果?有没有反直觉的发现?

## Step 1:素材考古

从原始材料里挖五类矿,列成素材清单:

| 矿 | 找什么 | 为什么值钱 |
|---|---|---|
| 硬数字 | 金额、耗时、数量、百分比、前后对比 | thread 的可信度锚点,一条没有数字的 thread 是散文 |
| 转折点 | "本来以为…结果…"的时刻 | 叙事的引擎 |
| 失败与代价 | 走错的路、白花的钱、丢掉的东西 | build-in-public 的信任货币——只晒赢的是广告 |
| 反直觉结论 | 和常识/教程/多数人做法相反的发现 | 转发的理由 |
| 可复用动作 | 读者今天就能抄的具体步骤/命令/清单 | 收藏的理由 |

每类至少挖 1 条;挖不出的类,标注"缺",并提示作者补(缺失本身可能意味着这段经历还不到发 thread 的时候)。

## Step 2:结构选型

按素材形态选骨架,只选一个:

1. **事故复盘型**(有失败与代价):平静开场 → 事发 → 最坏时刻 → 处置 → 数字结果 → 教训清单
2. **决策倒带型**(有转折点):结论先行 → 倒回决策现场 → 当时的选项 → 为什么选了这个 → 现在回看
3. **账本公开型**(有硬数字):数字全亮 → 拆构成 → 最意外的一项 → 下阶段目标
4. **方法沉淀型**(有可复用动作):痛点场景 → 我的做法逐步拆 → 每步的坑 → 完整清单收尾
5. **弧线叙事型**(时间跨度长):起点状态 → 三次关键转向 → 现在的状态 → 一句话总结这段路

判断依据写一行:"选 X 型,因为素材里最强的是____"。

## Step 3:逐条装配

输出 tweet-by-tweet 骨架,每条包含:

```
T3 ─────────────────────
[事实] 那天发现 /tmp 下有挖矿进程,htop 一看 CPU 拉满,多台机器全中
[你的话:____] ← 这里读者在等:你发现那一刻的第一反应
[配图位] htop 截图(如有)
```

装配纪律:
- 每条 ≤ 240 字符(英文)/ ≤ 100 字(中文),超了拆条
- T1 是 hook(见 Step 4),T2 必须兑现 T1 的承诺,不许拖到 T3
- 数字尽量前置到条首
- 全线 7-12 条;素材撑不起 7 条就建议改发单推,不硬拉
- 最后一条:总结 + 一个开放问题或下一步预告(给回复留钩子)

## Step 4:Hook 候选

给 T1 出 3-5 个候选,分不同杠杆:

- 数字杠杆:最狠的数字放第一句
- 损失杠杆:先说代价("这个错误花了我 X")
- 反差杠杆:常识 vs 我的发现
- 悬念杠杆:结果先行,过程后置

每个候选标注它赌的是什么心理。**不替作者选**,排序交给作者。

## Step 5:发布备注

- 建议发布时段(按目标受众时区)
- 需要准备的配图清单(截图/图表,标注哪条配哪张)
- 一行提醒:哪条最可能被断章取义,发布前自查

## 质量自检

- 素材清单五类齐不齐?缺的有没有明示?
- 骨架里有没有溜进去任何一句替作者写的观点?有 → 删掉换留白
- 把所有 [事实] 连起来读:一个陌生人能不能看懂这件事的始末?不能 → 骨架有洞
- 这条 thread 发出去,最好的结果是什么?(涨粉/引流/立专业形象/引发讨论)——和作者确认过再定稿
