# 归档纪律

执行归档(🟡)时照这份做,别自己发明格式。归档是可逆动作,但也要留下「为什么、什么时候」的痕迹,否则半年后没人记得这仓为什么在 `_archive` 里。

这份规范源自一次项目整理的实际做法:项目按类别归位、停掉的线进 `_archive/`、每条线留一份归档记录说明停用原因。

## 归档目录

- 唯一归档区:`~/projects/_archive/`
- 项目整体移过去,**保留原目录名**:`~/projects/apps/foo` → `~/projects/_archive/foo`
- 如果 `_archive` 里已经有归档过的老项目,照它们既有的摆法保持一致。

## 归档四步

**1. 归档前体检**
```bash
git -C "<项目>" status --short        # 有没有未提交改动
git -C "<项目>" log -1 --format='%cd' --date=short
git -C "<项目>" remote -v             # 有没有远端兜底
```
- 有未提交改动 → 先提醒用户 commit 或明确说「放弃这些改动」。不替他决定。
- 没有远端 + 有价值 → 提醒:归档只是移到本地另一个目录,不等于备份。

**2. 移动**
```bash
mv "~/projects/<类别>/<项目>" "~/projects/_archive/<项目>"
```
移之前最后确认一次目标路径存在、名实相符(别把活项目误当沉睡的移走)。

**3. 在项目里留一张归档卡**

在归档后的项目根目录写一个 `ARCHIVED.md`,内容极简:
```markdown
# ARCHIVED 2026-07-12

归档原因:<一句话,为什么停这条线>
末次活动:<last-commit 日期>
恢复方式:mv 回 ~/projects/<类别>/<项目> 即可;依赖需重装。
关联记忆:<如果记忆里有对应笔记,写文件名>
```
日期用当天真实日期,别写死。

**4. 记忆补一笔(值得的话)**

如果归档的是一条真业务线(不是随手停的小实验),提示用户在长期记忆里留一份 `<项目>-archived.md`,和其它归档记录一个规格:停用日期、原因、数据是否保留、能否恢复。这一步是提示用户做,或在用户明确授权后由记忆管理流程做——不擅自写用户的记忆库。

## 软链与依赖:动手前查一眼

归档会改变路径,任何指向老路径的东西都会断。移动前查:

```bash
# 有没有别处软链到这个项目
find ~/projects ~/.claude ~/.agents -maxdepth 4 -type l -lname "*<项目>*" 2>/dev/null
```
- 有软链指进来 → 报告用户,归档后这些链会失效,问要不要一起处理。
- 项目自己往外软链(如 skill 仓)→ 归档不影响别人,但记一笔。

## 归档 vs 删除

- **归档默认可逆**:`mv` 回去 + 重装依赖就能复活。所以拿不准一律归档。
- 归档一个项目时,可以顺手把它体积大头的构建产物(node_modules 等,见白名单)清掉再移——归档区不需要留可重建产物,能省不少空间。但这仍是一次删除动作,同样要进确认清单,不偷偷做。
- **绝不在归档时删源码/数据**:归档是把整个项目连源码一起搬走,不是就地删东西。
