# 可安全删除白名单 & 永不删黑名单

执行删除时才查这份。判断某个目录/文件能不能进删除清单,先过黑名单(命中即拒),再对白名单。**白名单之外、又拿不准的,一律不删——归档或保留。**

## 白名单:可重建构建产物

这些删了不丢信息,一条命令能重建。列进删除清单默认安全。

| 目录/文件 | 属于 | 重建方式 |
|---|---|---|
| `node_modules/` | npm/pnpm/yarn | `npm i` / `pnpm i` / `yarn` |
| `.next/` | Next.js | `next build` / `next dev` |
| `.nuxt/` | Nuxt | `nuxt build` |
| `.svelte-kit/` | SvelteKit | `vite build` |
| `dist/` `build/` `out/` | 通用打包产物 | 对应 build 命令 |
| `.turbo/` `.vite/` `.parcel-cache/` | 构建工具缓存 | 下次构建自动生成 |
| `.cache/` | 通用缓存 | 自动重建 |
| `target/` | Rust/Cargo | `cargo build` |
| `__pycache__/` `.pytest_cache/` `.mypy_cache/` `.ruff_cache/` | Python | 自动重建 |
| `coverage/` `.nyc_output/` | 测试覆盖率报告 | 重跑测试 |
| `.DS_Store` | macOS | 系统自动生成的垃圾 |
| `*.log`(构建/运行日志) | 日志 | 看一眼确认不是要留档的再删 |

## 灰名单:能删但要多问一句

技术上可重建,但删之前值得停一秒——里面**可能**混着不可重建的东西。

- **`.venv/` `venv/` `env/`** — Python 虚拟环境,一般 `pip install -r` 可重建。但确认里面没有手动装的本地包 / 编译好的扩展。
- **`node_modules/` 内含 patch** — 项目若用 `patch-package`(看 `patches/` 目录或 `postinstall`),补丁本身在源码里,删 `node_modules` 后 `install` 会自动重打,安全;但要确认补丁源文件在版本控制里。
- **`.git/`** — 技术上「可重建」需要 remote 完好。**默认按黑名单处理**(见下),只有在确认有完整远端且用户明确要删本地仓时才碰。
- **大的媒体/构建 `dist` 里混了手改文件** — 如果 `dist/` 被手动编辑过(没有对应源码),那它不是纯产物,退回黑名单。

灰名单项:不确定就不删,归档整个项目更安全。

## 黑名单:永不进删除清单

命中任一特征,这个目标**永远不进删除清单**。就算用户说「全删」,也要把它单独拎出来复述确认,而不是跟着白名单一起删。

- **源码**:`src/`、任何 `.ts/.js/.py/.go/.rs/.vue/.svelte` 等手写文件
- **数据**:`data/`、`*.db` `*.sqlite`、数据库 dump、导出的 CSV/JSON 数据集、`uploads/`、用户产出的素材/图片
- **配置与密钥**:`.env*`、`*.pem` `*.key` `*.crt`、`config/`、`secrets*`、`credentials*`、`.ssh/`、任何看起来像 token/口令的文件
- **版本历史**:`.git/`(删了丢掉全部提交历史,不可逆)
- **文档与账本**:`README`、`*.md` 笔记、记忆文件、归档记录本身
- **锁文件与清单**:`package.json`、`*.lock`、`Cargo.toml`、`requirements.txt`——这些是「怎么重建」的说明书,删了白名单里的东西就没法重建了

## 删前自检四问

对每个待删项,动手前默答一遍:

1. **它删了能一条命令重建吗?** 不能 → 不删。
2. **它名实相符吗?** 一个叫 `node_modules` 的目录,`ls` 一眼是不是真的依赖包?混着别的东西 → 停,报告。
3. **有没有正在用它的进程?** 项目此刻跑着 dev server → 先停服务或跳过。
4. **它是不是某条线上业务的一部分?** 生产站的 `dist`、正在服务的部署产物 → 不是「清理」是「拆线上」,不碰。

四问全过才执行。任一存疑,退回归档或保留。
