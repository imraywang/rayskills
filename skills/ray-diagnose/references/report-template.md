# 诊断报告模板

诊断阶段的标准交付物。给管理层看的,不是给自己看的——每一节都要为"帮客户做对的决定"服务。

## 结构(固定九节)

```
0. 执行摘要        一句话结论 + 三个关键发现 + 请拍板的 N 件事
1. 评估任务与方法   被请来回答的问题(原文) + 证据规模 + 方法(GROUND 六维)
2. 现状画像        六维雷达 + 成熟度等级 + 同业位置感("在中型零售里属典型状态")
3. 认知与现实差距   管理层认知 vs 实查结果 的对照表 + 证据编号——先校准事实再谈判断
4. 真问题          需求考古的对照陈述(陈述需求→考古发现→重构真问题→关系判定)
5. 病灶与机理      命中模式(按危险度排序) + 证据链 + 模式叠加的具体业务含义
6. 风险推演        如果按陈述需求直接干,6 个月后最可能的剧本(具体到场景和月份)
7. 可行性评级      红/黄/绿 + 触发的 gating + 变绿条件(可验收、给时限)
8. 建议路线概览    分期路线图(细节转 ray-proposal);今天请拍板的具体事项
```

## 执行摘要写法

- **一句话结论**必须同时给出:成熟度等级 + 可行性评级 + 一个反转("问题在评估暴露比上线暴露便宜十倍")。给坏消息一个建设性的框
- **关键发现**限 3 个,每个是"现象 + 数字 + 一句话机理",不堆砌
- **请拍板事项**限 3-5 件,具体到可签字(批预算/承诺编制/授权某机制),不是"加强重视"

## 三条写作纪律(全文贯穿)

1. **每个论断挂证据编号**。表格里的每个数字后标 `(R4、T5)` 之类来源。被质询时 10 秒内回溯
2. **坏消息配出路**。任何"不行/红灯/缺失"后面紧跟"……的条件是/建议这样补"。红灯的价值在给变绿路径
3. **保护信息源**。证据引用不指向个人("业务访谈中多位受访者提到"),尤其与高管认知相反的证词。匿名承诺是信用资产

## 风险推演写法(第 6 节)

这一节最有说服力,因为它让客户看见"不听劝的下场"。写法:
- 假设跳过报告直接采购,按月份推演(第 1-2 月 demo 效果好 → 第 3-4 月一线弃用 → 第 5-6 月对外上线触雷)
- 点破:剧本里**没有任何技术失败,失败全部来自知识供给侧**——这就是判红灯的含义
- 若客户举同行案例反驳,补一句:行业演示≠行业落地,公开可见案例多是"窄场景+白名单语料"形态,恰是你为他设计的 Phase 0

## 语气

- 你是站在客户长期利益一边的人,不是推销员。红灯是专业资本不是拒绝
- 认知差不是谁的过错,是企业从 300 人长到 800 人时知识管理没同步升级的自然结果——这样说,校准了事实又不伤人
- 数据自己会说话。用采样质检的客观数字呈现幻觉,而不是评判"你们做得差"
