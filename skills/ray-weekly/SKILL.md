# ray-weekly:每周复盘

把「每周人肉把所有项目和数据盘一遍」这件重复劳动固化下来。这个 skill 针对的场景很具体——Ray 每周要手动 `cd` 进十几个仓看 git log、翻 X 数据、在脑子里对三条现金流线各自到哪了,盘完还常常盘出一堆流水账却没结论。本 skill 替掉的就是这段扫描 + 判断的手工活。

**你的立场**:盘点是为了**判断**,不是为了罗列。列 20 条 commit 谁都会,值钱的是那句「这周真正推进的只有 Relay 计费这一件,其余都是维护性动作」。最终产出是决策输入,不是活动日志。

## 用法

```
/ray-weekly                    # 默认盘最近 7 天(上个周一至今)
/ray-weekly 2026-07-01..07-07  # 指定日期范围
/ray-weekly 上周                # 自然语言范围也认
```

先确定时间窗口(默认「1 week ago」到现在),四步走。项目分类地图和现成扫描命令 → **读 [references/project-map.md](references/project-map.md)**,不要凭记忆敲仓路径。

## Step 1:项目动态——动过的 vs 沉睡的

扫 `~/projects/<类别>/` 各仓本周提交,把仓分三档:

- **动过**:窗口内有 commit。记下提交数 + 最后一次提交在干嘛(看 message,不是数数量)。
- **沉睡**:窗口内零提交,但不是归档项目。这类要留意——上周还在推、这周停了,是信号。
- **归档**:`_archive/` 下的一律排除,不计入「沉睡」制造焦虑。归档 = 主动停,不是停滞。

⚠️ 关键区分:**沉睡 ≠ 归档**。`_archive/` 里的仓再久没动也正常,别把它们列进「本周停滞」。反过来,一个活跃业务仓突然一周没动,才值得单独标出来问一句「为什么」。

扫描命令(按类别批量跑 git log、列各仓最后提交时间)在 references/project-map.md。dist 产物、node_modules、非 git 目录自动跳过。

## Step 2:内容 / IP 数据

拉本周对外内容的量化面:

- **X 数据**:优先调 `/ray-metrics`(若已就绪)拉本周涨粉、曝光、互动、爆款推文;没有就让 Ray 给或跳过,不硬编数字。
- **其它平台**:公众号、飞书 IP 库、小红书等有数据的一并纳入。
- 数据看**趋势和异常**,不是搬台账:哪条内容跑出来了、涨粉是加速还是平了、有没有断更。

**账本纪律**:所有数字以 Ray 提供或工具实拉为准。**不臆造、不估算、不「大概涨了几百粉」**。拉不到就写「本周 X 数据未拉取」,留白比编造强。这条是硬线——Ray 的账本不容猜。

## Step 3:三条现金流线状态

对每条线判断本周**有无实质推进**(不是「有没有想到」,是「有没有动作落地」):

| 线 | 本周看什么算推进 | 对应项目/信号 |
|---|---|---|
| **Upthos Relay + 引流工具** | 新增付费/续费、计费或部署改动、引流工具(如 ai-ip-check)带量 | `upthos/relay`、引流类仓的 commit + Ray 报的客户动态 |
| **企业咨询** | 新线索、诊断/方案交付、回款、案子推进 | `/ray-diagnose`·`/ray-proposal` 的实际使用、Ray 报的咨询进展 |
| **IP 增长** | 内容产出与数据、影响力动作、装配稿落地 | `content/x-pipeline`、Step 2 的数据、飞书 IP 库更新 |

每条线给一句话结论 + 一个方向标记(推进 ↑ / 持平 → / 停滞 ↓)。

**诚实优先**:某条线本周真没动,就写「停滞 ↓,本周无动作」,**不要从边角料里硬抠出「进展」来凑三条都亮**。一条线停了本身就是最有价值的信号——它告诉 Ray 注意力这周漏了哪。硬找进展等于自欺,直接毁掉这个 skill 的用处。

## Step 4:综合收口

这是唯一真正动脑的一步,前三步都是为它备料。产出四段:

1. **本周真正动了什么**(≤5 条,判断不是流水账):把一堆 commit 收敛成几句结论。「Relay 上线阶梯计费」是一条;「修了 3 个 typo、更新 README」合并成「维护性动作」一句带过或不写。区分**推进性**动作和**维护性**动作。
2. **哪条线在推进,哪条在停**:直接引用 Step 3 的三个标记,点出本周注意力实际落在了哪、漏了哪。
3. **下周聚焦(≤3 件)**:少而实。每件是可动手的具体动作,不是「多关注 IP」这种口号。**宁可只写 1 件真能推的,也不凑 3 件稀释掉。** 贪多 = 下周又一事无成。聚焦来自「本周停滞的线 + 有势能该加注的线」,不是把没做完的全搬过来。
4. **值得记一笔的**(可选):一次性的教训、意外的信号、需要 Ray 拍板的岔路口。

## 输出形态

默认输出一段结构化文字(四段收口在最前,项目/数据/现金流明细在后供回溯),终端直接可读。Ray 要归档成文档或长图再另说。

**若 Ray 要把复盘转成对外发布的「本周进展」帖**:那越过了功能性产出的边界,进入人设声音。此时**只交装配骨架 + 素材**(哪几件事值得对外讲、数据点、时间线),观点和口吻留 `【你的话:____】` 空位给 Ray 自己填。内部复盘可以全交付,对外发声不代笔。

## 纪律

- **判断 > 罗列**:交结论不交 commit 日志。盘出「没结论的流水账」等于没盘。
- **账本以 Ray 为准**:数字实拉或 Ray 给,拉不到就留白,绝不臆造、不质疑 Ray 报的数。
- **诚实标停滞**:线停了就说停,不硬凑进展。停滞是信号不是失败。
- **聚焦≤3 且实**:少而实,可动手。凑数的聚焦不如不写。
- **归档不算停滞**:`_archive/` 排除在动态盘点外。

## 什么时候不该用

- 要深挖某一个项目的进展 → 直接进那个仓,别起全局盘点。
- 要写对外发布的进展帖 → 本 skill 只到骨架,血肉走不代笔。
- 跨季度 / 战略级复盘 → 那是更大尺度的方向校准,不是一周动静盘点。
