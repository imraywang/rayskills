# references/handoff.md —— 交接与维护(B2B 站收尾)

站上线不是活干完。B2B 站的真正终点是**客户能自己维护**——否则每改一个字都来找你,你把自己变成了客户站的人肉运维。这份交接物设计源自 FOHO 客户维护手册,可直接作为 B2B 交付模板。

## 绿 / 黄 / 红三区心智模型

给客户一套"什么能自己动、什么照模板动、什么别碰找开发"的分区,是防止客户手滑搞坏站的最简心智模型:

| 区 | 含什么 | 客户能做什么 |
|---|---|---|
| 🟢 **绿区(随便改)** | 产品、页面文字、FAQ、Logo、菜单 | 放手自己改,改坏了影响也小 |
| 🟡 **黄区(照模板改)** | HTML 块 | **只改引号内的文字**,别动标签结构 |
| 🔴 **红区(别动,找开发)** | 全局样式(Site Settings)、模板结构(Theme Builder)、CSS Classes、代码 Snippets | 一律别碰,要改找开发 |

## 落地手段(把三区变成客户真能用的东西)

- **建受限编辑角色**(如 Members 插件):给客户账号**只开**绿区权限——产品编辑 + 页面文字编辑 + 看询盘提交,关掉全局设置/模板/代码入口。物理上够不着红区,比嘴上叮嘱靠谱。
- **录 10 分钟操作视频**,覆盖三个高频场景:加一个产品 / 改一段文字 / 看询盘。视频比文档管用。
- **导出 PDF 交客户**:🟢🟡 的操作手册交客户,🔴 部分留开发内部。

## 询盘双通道(防漏单,B2B 站的钱在这)

询盘漏一单 = 漏一笔生意。**别只靠一条通道**:

1. **邮箱**:提交即发到客户收件箱(实时看)
2. **后台存档**:表单提交同时落库到后台(如编辑器的 Submissions),客户每周看一次兜底

两条通道任一挂了,另一条还在。上线验证时两条都要实测通(见 `switch-runbook.md` 表单验证项)。

## 交接前的清理与安全项

- 撤销上线期临时开的应用密码 / 卸载临时集成工具——别把施工期的后门留在客户站上
- 关掉重复的 SEO 自动项(如 JSON-LD 双份里插件自带那份)
- 确认联系方式所有出现处(页头/页脚/Contact 卡/浮窗/产品页按钮/表单收件)都是最终值——散落多处,别漏
- 把回滚方式和线上关键凭证位置(不打印明文,只给存放位置)写进交接文档

## 交接清单模板(输出给客户/自己存档)

```
站点:        <域名>            托管:<平台>
路线:        <纯静态 / headless / 全栈>
内容维护:    <谁> 通过 <后台入口>,权限=绿区
询盘去向:    邮箱 <收件箱> + 后台 Submissions(每周看)
SEO:         sitemap 已交 GSC,URL 结构 <路径清单>
凭证:        后台/DNS/邮箱凭证 → <存放位置>(不打印明文)
回滚:        DNS 改回 <旧IP> + <旧平台> 重绑域名
维护手册:    🟢🟡 → 客户PDF+视频   🔴 → 开发内部
待办:        [ ] 观察期404补301  [ ] 撤临时密钥  [ ] TTL调回
```
