# Ray 长文生产管线

用于用户明确要求“把这个 idea 做成文章并送进公众号或 X 后台”“走完整条内容管线”或“把内容生产流水化”时。正常步骤连续执行；只有事实边界、核心判断、真人形象、登录或线上写入授权需要用户介入。

## 完成状态

以下条件同时满足才算完成：

- 本地长文通过事实、语气、情绪和移动端可浏览性检查。
- 公众号、普通 X 和所需的 X Article 封面按各自比例生成并实际查看。
- 用户要求公众号草稿时：本地富文本预览已确认；微信后台只更新原草稿或明确新建一份；回读的标题、摘要、封面、正文、署名和编码正确。
- 用户要求 X Article 时：X 后台只存在一份对应草稿，标题、正文、封面与本地版本一致。
- 所需平台的草稿状态、封面路径和内容指纹已回写。
- 未发布文章。

## 阶段与交接

### 0. `ray-obsidian`：准备本地内容地基（按需）

用户已有兼容知识库时跳过。没有知识库、但要求长期内容生产和知识回流时，先确定根目录，安全建立资料、知识、成稿包、调研、草稿、发布与素材结构。已有文件不得覆盖或移动；最终检查必须为 `ready`。

### 1. `ray-writer`：从材料到可读长文

输入：idea、剪藏、调研、已有草稿或用户真实经历。

必须交付：

- 正文、成稿包和事实清单。
- 一句话判断、核心冲突句、情绪曲线和传播句。
- 适合移动端浏览的结构：长文使用具体的二级标题作为阅读锚点；只加粗完整的关键判断；段落之间没有多余空行。

事实或作者经历不清时停在这一阶段；不能用流畅文字掩盖证据缺口。

### 2. `ray-cover`：从判断到平台封面

输入：完整正文、成稿包、一句话判断、核心冲突和传播句，不能只传标题。

必须交付：

- 一个清楚的编辑隐喻。
- 公众号、普通 X，以及需要时的 5:2 X Article 封面：默认 Image 2 一次生成画面与中文并逐字检查；出现错字、频繁改题或跨平台漂移时回退无字底图 + 确定性排版。
- 逐字正确、缩略图可读、标题没有被裁切的成品和清单。

各平台共享视觉母题，但不能拉伸同一张成品；兜底模式下图片模型不负责中文排版。

### 3. `ray-wechat`：从本地成稿到公众号草稿（按需）

输入：通过检查的 Markdown、公众号封面、署名策略和用户确认过的排版方向。

必须交付：

- 干净正文 HTML 和带复制按钮的手机端预览；没有空段落、英文拆词或署名占位。
- 用户确认后优先更新 frontmatter 中记录的原 media ID，没有 ID 且明确授权时才创建。
- 更新旧草稿时保留封面、作者和评论设置；写入后强制 UTF-8 回读。
- 核对标题、摘要、正文全部原文块、章节、重点、空段落和署名，再回写草稿状态。

排版预览确认与线上草稿写入是两道独立门。接口成功码不能代替回读验收；本 Skill 不发布文章。

### 4. `ray-x-article`：从本地成稿到可恢复草稿（按需）

输入：通过检查的 Markdown、5:2 封面和交付包。

必须交付：

- 优先续写本地已记录的草稿 URL；没有 URL 才在 Drafts 按标题查重。
- 使用富文本粘贴保留二级标题和加粗，不把 Markdown 空行变成空白段落。
- 完成封面裁切确认、正文结构核对、Preview 检查和返回编辑页后的保存确认。
- 回写 `x_article_status: draft`、草稿 URL、封面路径、保存时间和内容指纹。

## 恢复规则

| 状态 | 处理 |
|---|---|
| 正文已写入，封面因浏览器本地文件权限失败 | 保留同一草稿，回写 `draft-needs-cover`；权限开启后从原 URL 续传，不新建 |
| 草稿 URL 已存在 | 直接打开并核对标题；不可访问时才回 Drafts 查找 |
| 本地已是 `draft`，但用户提到过去上传失败 | 先核验后台当前状态；封面实际缺失才补传，不能按旧叙述覆盖 |
| 富文本粘贴失败 | 不接受带 Markdown 符号或多余空行的结果；恢复格式后重新核对数量 |
| 封面上传后出现 `Edit media` | 保持默认 5:2 构图，点击 `Apply`，再以封面显示和 `Remove photo` 为成功信号 |
| 保存状态没有更新 | 留在原草稿，等待可见保存状态；不能把“已经输入”当成“已经保存” |
| X 界面文字或结构变化 | 重新读取当前页面，以可见控件为准；不盲点坐标，不创建替代草稿 |
| 公众号文章已有 `wechat_draft_media_id` | 先读取并更新原草稿；不得调用新建接口制造重复项 |
| 公众号回读出现 `Ã`、`å` 等乱码 | 使用本地正确 HTML、frontmatter 标题和明确摘要覆盖同一草稿，强制 UTF-8 回读全文 |
| 公众号排版器自动加入署名或三连 | 按用户署名策略整块删除，重新生成预览和正文，再更新原草稿 |
| 公众号接口写入成功但正文数量不符 | 视为失败；比对全部原文块、章节、空段落和重点数量后原地修复 |

## 最终验收

逐项核对：

1. 所需平台标题完全一致。
2. 正文首句、末句和全部原文块一致。
3. 空白段落为零；章节和重点数量与本地交付包一致。
4. 公众号封面 ID 保持正确；X 封面在编辑页和 Preview 都完整显示。
5. 公众号回读没有中文乱码，署名策略生效；X Preview 中作者、标题、段落和结尾正常。
6. 所需平台的最近保存状态可验证。
7. 本地草稿状态已更新为 `draft`，中断状态已清除。
8. 没有发布文章。
