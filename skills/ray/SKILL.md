# ray:主入口与路由

rayskills 是 Ray 在真实业务里沉淀的 builder 工具箱。用户不需要记住十几个 skill 名——把真实处境交给 `/ray`,由它判断此刻最该做的一步,分发到对的成员 skill。

**路由哲学(单步动态导航)**:每次只决定**当前这一步**,不预先排一条固定的长链。分发到一个 skill、拿到结果后,用户再回到 `/ray`,结合新结果和反馈判断下一步。方向由实际结果决定,不由剧本决定。

## 怎么分发

1. 读当前对话已有的信息(处境、目标、材料、卡点)。
2. 对照下面的路由表,判断意图落在哪条线、哪个 skill。
3. **只选一个 skill**,说明为什么是它,然后按那个 skill 的 SKILL.md 执行。若信息不足以判断,先问一个最关键的澄清问题,不硬猜。
4. 完成后提示用户:可以再 `/ray` 让我基于结果决定下一步。

## 路由表

| 巡检代理节点/中转链健康、核订阅 | `ray-nodecheck` |

### 内容 / IP

|---|---|

## 常见衔接(供判断下一步)

这些不是固定流水线,是"上一个结果出来后,通常接哪个"的经验参考:

- `ray-diagnose` 出了诊断结论 → 通常接 `ray-proposal` 出方案;若结论是红灯,方案的 Phase 0 就是补齐前提。
- `ray-launch` 上线了一个站 → 可接 `ray-thread` 把上线过程做成 build-in-public。
- 任何一段实战(开荒/巡检/事故处置)完成 → 可接 `ray-thread` 或 `ray-tweet` 把它变成内容。
- `ray-benchmark` 拆完对标 → 可接 `ray-anvil`(锻造差异化产品)或 `ray-thread`(把洞察写出来)。
- `ray-metrics` 出了数据规律 → 喂给 `ray-weekly` 的内容数据节,或指导 `ray-tweet` 下一批方向。

## 边界与纪律

- **只做路由,不替代成员 skill 的判断**。选定后交给那个 skill 的完整流程,别在这里半路自己干。
- **内容类守不代笔**:分发到 `ray-thread`/`ray-tweet` 时,产出是骨架+素材,观点血肉留给 Ray 自己填。
- **私有 skill**(ray-proposal / ray-weekly / ray-cleanup)不随开源发布;路由可用,但对外分享的工具箱版本里它们不出现。
- 一个请求可能同时沾几条线——选**此刻最该做的那一步**,把其余的作为"下一步"提示,而不是一次全上。
