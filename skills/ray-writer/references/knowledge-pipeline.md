# Ray 本地知识库写作管道

下列路径都相对于已确认的知识库根目录 `<vault>`。兼容知识库应包含 `.ray-obsidian.json`，或至少同时存在 `10-创作/`、`20-知识/`、`30-资料/` 与 `50-系统/`。不要依赖知识库名称。

## 文件职责

- `10-创作/10-灵感/10-待评估/`：等待人判断是否值得推进的选题与审核卡。
- `10-创作/10-灵感/20-成稿包/`：已经决定推进，等待调研或写作的主题装配单。
- `30-资料/10-自主调研/<主题>/`：事实清单、来源索引、数据快照和结构研究。
- `10-创作/20-草稿/`：当前有效的完整草稿。
- `40-发布/`：用户确认后的发布版本。
- `60-素材/10-图片/10-封面/<YYYY-MM-DD-主题>/`：封面任务包、无字底图、公众号与 X 成品和清单。
- `20-知识/`：发布或研究过程中形成的长期观点、案例和方法。

## 正常流转

外部资料先进入资料区并生成审核卡。人工选择“仅保留为写作素材”后，系统自动建立成稿包，而不是修改原始资料。

当用户直接从一个已经沉淀为长期知识的角度发起写作时，也建立新的成稿包，并同时链接长期知识、审核卡和原始来源。

写作任务继续推进时：

1. 丰富成稿包中的问题、判断、情绪和真实材料。
2. 建立主题调研目录和事实清单。
3. 生成一份当前草稿。
4. 完成检查和用户确认。
5. 需要封面时，把正文、成稿包和一句话判断交给 `ray-cover`，生成双平台素材。
6. 用户明确要求发布或归档时，才写入发布区。
7. 将新增的长期认识回写为独立知识，不把整篇文章复制进知识区。

## 版本规则

- 原始资料保持原样。
- 审核卡只记录判断、动作和结果文件。
- 成稿包保存写作依据，不塞入完整正文。
- 草稿目录每个主题只保留一份当前有效版本；需要保留历史时依赖版本记录或移入归档。
- 用户个人笔记和明确手写内容永远保留。

## 命名

- 成稿包：`YYYY-MM-DD-主题.md`
- 调研目录：`YYYY-MM-DD-主题/`
- 草稿：`YYYY-MM-DD-标题.md`
- 文件名使用读者能理解的中文主题，不使用随机编号代替语义。

## 最小触发方式

用户只需说“把这个 idea 写成文章”并指出资料、审核卡或主题。系统负责发现上下游材料、创建缺少的文件和完成检查。若没有兼容知识库，完整管线先交给 `ray-obsidian` 初始化；用户只要一次性文章时可以在当前工作区降级交付，但必须明确不会自动回流长期知识。
