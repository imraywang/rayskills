# Ray 本地知识库结构

## 设计目标

这套结构服务于两个循环：资料逐步变成可复用知识；知识与真实材料被装配成内容并在发布后回流。它不是百科全书，也不要求一次整理完历史笔记。

## 目录职责

| 目录 | 职责 |
|---|---|
| `00-入口/10-工作台/` | 使用说明、当前入口与人工检查点 |
| `10-创作/10-灵感/10-待评估/` | 尚未决定是否推进的选题与审核卡 |
| `10-创作/10-灵感/20-成稿包/` | 已决定推进的写作任务包 |
| `10-创作/20-草稿/` | 每个主题当前有效的完整草稿 |
| `20-知识/` | 可长期复用的概念、观点、案例、方法与地图 |
| `30-资料/` | 外部原文、剪藏、书签和自主调研，不与长期知识混放 |
| `40-发布/` | 用户确认后的发布版本，按平台分目录 |
| `50-系统/` | 设计说明、流程和模板 |
| `60-素材/10-图片/10-封面/` | 封面任务包、底图和平台成品 |
| `70-思考/` | 尚不适合变成正式知识或内容的个人思考 |
| `90-归档/` | 失效但仍需保留的内容；不作为删除区 |

## 知识分类

长期知识使用 frontmatter 的 `kind` 作为分类真源：

- `concept`：概念
- `entity`：人物或组织
- `question`：值得持续回答的问题
- `viewpoint`：可辩论的判断
- `case`：案例
- `playbook`：方法与操作方案
- `map`：主题地图
- `guide`：导航与系统说明

目录只是浏览便利。不要因为移动文件而批量改写知识类型。

## 最小内容流

1. 原始输入进入 `30-资料/`，保持来源与链接。
2. 值得长期复用的部分提炼为 `20-知识/` 的独立笔记。
3. 值得写成内容的主题建立成稿包，先确认问题、判断、事实边界、个人材料与情绪曲线。
4. 调研进入 `30-资料/10-自主调研/<主题>/`，完整正文进入 `10-创作/20-草稿/`。
5. 用户确认发布后才进入 `40-发布/`。
6. 发布形成的新观点、案例和方法回流为独立知识，而不是把整篇文章复制到知识区。

## 命名与版本

- 成稿包：`YYYY-MM-DD-主题.md`
- 调研目录：`YYYY-MM-DD-主题/`
- 草稿：`YYYY-MM-DD-标题.md`
- 文件名使用有意义的人类语言，不用随机编号代替主题。
- 同一主题只保留一份当前草稿；历史版本交给 Git、同步服务或归档目录处理。
