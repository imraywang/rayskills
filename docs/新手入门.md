# rayskills 新手入门

rayskills 是一套包含 14 个 Skill 的 builder 实战工具箱。它不只给建议，而是按真实工作流程完成下一步、验证结果，并在中断后从原状态继续。

## 第一次怎么用

不用记 Skill 名，把真实处境交给 `/ray`：

```text
/ray 我有个客户想上 AI 客服，不知道现在能不能做
/ray 上周把被入侵的服务器清干净了，想把过程写出来
/ray 帮我把这个 Next 站上线到 Vercel
/ray 把这个 idea 写成文章、做封面，再保存到公众号和 X Articles 草稿箱，不要发布
/ray 我没有 Obsidian 知识库，先帮我搭好再开始写作
```

`/ray` 有两种工作方式：

- 目标还不稳定时，只选择当前最该做的一步。
- 用户已经明确终点、阶段之间有正式交接时，连续执行整条管线。

普通阶段交接不会反复要求确认；发布、删除、生产修改和无法推断的事实边界仍会停下来确认。

## 熟悉后直接调用

```text
/ray-writer 把这个 idea 写成一篇有事实和传播力的公众号长文
/ray-writer 把这篇定稿提炼成一条 2 到 3 分钟的口播视频稿
/ray-obsidian 在这个本地目录搭一套后续能接 ray-writer 的知识库
/ray-cover 给这篇定稿文章做公众号、普通 X 和 X Article 封面
/ray-wechat 把这篇定稿排版并更新到原公众号草稿，先预览再写入
/ray-x-article 把文章和 5:2 封面保存到已有 X 草稿，不要发布
/ray-multimodel 让 Grok 和 Claude 独立给方案，由当前会话验收
/ray-multimodel scout quick 查过去 24 小时 X 上关于这个产品的真实反馈
/ray-consult 一家 800 人化妆品公司想上 AI 客服
/ray-writer 把 ~/projects/xxx/RETRO.md 这段实战做成 thread 骨架
/ray-vps root@1.2.3.4
/ray-vps check
```

## 长文生产管线

用户明确要求从 idea 一路做到公众号或 X 草稿时，`/ray` 会连续执行：

0. `ray-obsidian`（按需）：没有兼容知识库时，先安全建立资料、知识、成稿包、草稿、发布与回流结构。
1. `ray-writer`：建立事实清单、文章判断、情绪结构和完整正文；检查二级标题、重点加粗与空行。
2. 需要口播时，`ray-writer` 从母稿重新选择一个判断，生成逐字稿、拍摄节奏和素材清单；需要拼贴画面再接 `ray-broll`。
3. `ray-cover`：提炼一个视觉隐喻，默认 Image 2 直出并逐字检查，必要时回退确定性排版，分别输出公众号、普通 X 和 5:2 X Article 封面。
4. `ray-wechat`（按需）：确认手机端富文本预览，优先更新原草稿，回读标题、封面、全文、署名和中文编码。
5. `ray-x-article`（按需）：优先恢复已有草稿，保留富文本格式，检查封面、首尾、预览和保存状态，停在草稿。

完整规则见 [Ray 长文生产管线](../skills/ray/references/content-pipeline.md)。

## Skill 全目录（14）

| Skill | 干什么 |
|---|---|
| `/ray` | 主入口与阶段编排，不知道用哪个就交给它 |
| `/ray-vps` | 开荒：加固、代理栈、验证与交接；`/ray-vps check` 巡检，只读不改 |
| `/ray-obsidian` | 新建、检查或增量适配本地 Obsidian 内容知识库 |
| `/ray-writer` | idea、资料或草稿 → 中文长文；实战 → thread 骨架；定稿 → 口播内容包 |
| `/ray-cover` | 定稿文章 → 公众号、普通 X、X Article 封面 |
| `/ray-broll` | 口播文稿 / 选题 → 拼贴 B-roll 或完整拼贴讲解片 |
| `/ray-wechat` | 定稿文章与公众号封面 → 已验证的微信公众号草稿 |
| `/ray-x-article` | 长文与 5:2 封面 → 已验证的 X Articles 草稿 |
| `/ray-metrics` | X 账号数据周报与传播规律 |
| `/ray-benchmark` | 对标拆解，判断可迁移性 |
| `/ray-report` | magazine 风格 HTML、PDF 与公众号报告 |
| `/ray-consult` | 诊断段：就绪度评估与红黄绿评级；方案段：蓝图、分期与预算 |
| `/ray-launch` | 落地页 / B2B 站上线全流程 |
| `/ray-multimodel` | Grok / Claude / Codex 分工、复核、竞赛；隔离检索 X、Reddit 与网页 |

成员之间怎么衔接见 [Skill 关系图](skill-link-map.md)。

## 设计原则

1. **方法论来自真实工作**：保留可迁移的规则、坑位和验收标准，不堆履历。
2. **能防错才有价值**：每个 Skill 都包含正常、边界和失败场景。
3. **明确终点就做完**：正常交接不断点；结果必须实际验证。
4. **不越权**：不虚构经历和数据，不自动发布；删除与生产修改守住确认边界。
5. **可以恢复**：优先读取已有状态、草稿 URL 和本地记录，不制造重复项。
